# tobii4c

Standalone Tobii Eye Tracker 4C support for Linux/X11.

`tobii4c` provides realtime gaze tracking, fullscreen calibration,
eye controlled mouse interaction and experimental geometry-aware
3D gaze processing for Linux desktops.

The project focuses on:

* accessibility
* assistive input
* low latency gaze interaction
* standalone Linux eye tracking
* experimental HCI
* geometry-aware gaze mapping
* realtime filtering and stabilization

Designed primarily for:

* hands-free desktop interaction
* accessibility workflows
* Linux eye tracking research
* assistive desktop environments
* experimental gaze interaction

Tested primarily on Ubuntu-based Linux systems using X11.

# Features

## Low latency gaze tracking

The tracker streams realtime gaze coordinates from the Tobii Eye Tracker 4C
with low latency optimized for direct desktop interaction.

Implemented pipeline stages:

* validity filtering
* stale frame rejection
* smoothing
* edge damping
* deadzone suppression
* drift reduction
* motion stabilization

Example:

```text
Look at a desktop icon → cursor follows instantly.
```

## Fullscreen X11 calibration

`eyecalib` provides a fullscreen X11 calibration environment.

Implemented features:

* fullscreen override-redirect overlay
* keyboard/pointer grab
* multi-point calibration
* edge calibration
* retry logic
* realtime gaze debugging
* invalid sample rejection
* local calibration persistence

Calibration targets use:

* concentric rings
* thin precision outlines
* center fixation point
* crosshair visualization

Targets are intentionally positioned near display edges
to improve fullscreen reachability.

## Tobii Geometry Bootstrap

`tobii4c` can optionally import geometry information from:

```text
~/.config/TobiiProEyeTrackerManager/db/da-setups.db
```

The file is generated by:

```text
TobiiProEyeTrackerManager
```

The database format is:

```text
JSON-lines / NeDB style
```

Each line contains one JSON geometry profile.

Example:

```json
{
  "name":"NucBox",
  "trackerPlacement":"under",
  "tracker":{"angle":20},
  "display":{
    "width":341.66,
    "height":192.18
  }
}
```

---

# Imported Geometry Data

Currently imported bootstrap values:

| Field            | Purpose                   |
| ---------------- | ------------------------- |
| trackerPlacement | vertical compensation     |
| tracker.angle    | screen tilt estimation    |
| display.width    | physical display geometry |
| display.height   | physical display geometry |
| displayArea      | 3D display plane          |
| guidePosition    | future center estimation  |

---

# trackerPlacement

Example:

```json
"trackerPlacement":"under"
```

Meaning:

The tracker is physically mounted below the display.

Practical consequences:

Trackers mounted below the screen often produce:

* upward gaze skew
* asymmetric vertical distortion
* reduced upper-edge precision

The runtime compensates this using:

```text
sensor_offset_y
```

This improves:

* upper edge reachability
* vertical stability
* fullscreen interaction

---

# tracker.angle

Example:

```json
"tracker":{"angle":20}
```

Meaning:

The tracker is mounted at approximately:

```text
20°
```

relative to the display plane.

Practical consequences:

Without correction:

* gaze vectors intersect the screen incorrectly
* vertical drift increases
* edge distortion becomes asymmetric

The runtime therefore imports:

```text
screen_tilt = 20°
```

This can later be used for:

* perspective correction
* 3D reprojection
* head compensated gaze mapping

---

# display.width / display.height

Example:

```json
"display":{
  "width":341.66,
  "height":192.18
}
```

Meaning:

Physical monitor size in millimeters.

The runtime can derive:

```text
mm_per_pixel
```

using:

```text
physical_width_mm / screen_resolution_x
```

Example:

```text
341.66 mm / 1366 px
≈ 0.25 mm/px
```

This enables:

* geometry-aware projection
* physically correct scaling
* future distance correction
* realistic 3D mapping

---

# displayArea

Example:

```json
"displayArea":{
  "bottomLeft":[-169.4,16.8,-1.9],
  "topLeft":[-169.4,197.4,63.8],
  "topRight":[172.2,197.4,63.8]
}
```

This defines the physical display plane in 3D space.

Coordinate meaning:

```text
X = horizontal
Y = vertical
Z = depth
```

relative to the tracker.

---

# Screen Plane Mathematics

The display plane is mathematically defined using vectors.

Example:

```text
A = bottomLeft
B = topLeft
C = topRight
```

Two plane vectors:

```text
u = B - A
v = C - A
```

The screen normal vector becomes:

```text
normal = u × v
```

(cross product)

The resulting normal vector describes:

```text
physical screen orientation in 3D space
```

---

# Practical Interpretation

Example:

```text
bottomLeft.z = -1.9
topLeft.z    = 63.8
```

Meaning:

```text
upper screen edge is physically farther away
than lower screen edge
```

Therefore:

```text
screen is tilted backwards
```

The runtime derives:

```text
screen_skew_z
```

Example:

```text
63.8 - (-1.9)
≈ 65.7
```

The current implementation converts this into:

```text
sensor_offset_y = -(screen_skew / 5000)
```

Example:

```text
-65.7 / 5000
≈ -0.0131
```

# Why The Tobii DB Is Only Bootstrap Data

The Tobii geometry database describes:

```text
physical installation geometry
```

but NOT:

* individual eye behavior
* posture
* seating position
* gaze drift
* user asymmetry
* lighting conditions

Therefore:

```text
geometry alone is insufficient
```

Actual runtime precision is determined primarily by:

* calibration quality
* runtime filtering
* mesh correction
* gaze stability

Architecture:

```text
Tobii DB
    │
    ▼
initial geometry bootstrap
    │
    ▼
runtime calibration mesh
    │
    ▼
final gaze mapping
```

# Calibration System

Calibration data is stored in:

```text
~/.local/tobii_4c/calx11.conf
```

Example:

```text
version 2

sensor_offset_x 0.000000
sensor_offset_y -0.013100

screen_distance 63.000
screen_tilt 20.000

gaze_smooth 0.120
cursor_smooth 0.220

edge_zone 0.080
edge_power 1.350

# calibration
0.523789 0.455566 0.500000 0.500000
```

---

# Calibration Point Mathematics

Each calibration point stores:

```text
raw_x raw_y target_x target_y
```

Meaning:

```text
measured gaze position
→ expected screen position
```

Example:

```text
0.523789 0.455566
→
0.500000 0.500000
```

Interpretation:

The tracker estimated:

```text
52.4% horizontal
45.5% vertical
```

while the actual target was:

```text
50%
50%
```

The runtime therefore derives a correction offset.

---

# Calibration Mesh

The 3x3 calibration creates a correction mesh:

```text
P00  P01  P02
P10  P11  P12
P20  P21  P22
```

The runtime later interpolates between these points.

This compensates:

* nonlinear distortion
* edge compression
* diagonal skew
* asymmetric tracking

---

# Why Eye Trackers Need Mesh Correction

Eye trackers are NOT linear devices.

Typical distortion:

```text
center:
    accurate

edges:
    compressed

corners:
    unstable
```

Without correction:

```text
user looks at corner
→ cursor stops before edge
```

The calibration mesh compensates this behavior.

---

# Adaptive Edge Reachability

Eye trackers frequently compress movement near display borders.

The runtime compensates this using nonlinear edge amplification.

Implemented runtime parameters:

```text
edge_zone
edge_power
```

Benefits:

* easier corner access
* reduced edge sticking
* improved fullscreen usability
* better menu interaction

---

# Mathematical Edge Scaling

Conceptually:

```text
if x < edge_zone:
    amplify(x)

if x > 1-edge_zone:
    amplify(x)
```

using nonlinear scaling:

```text
pow(distance, edge_power)
```

This improves:

* fullscreen usability
* corner access
* small UI targets
* edge interaction stability
